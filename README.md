# MicroStore - Django Microservices on Kubernetes

MicroStore is a small e-commerce platform built as independent Django services and packaged for Kubernetes. The project demonstrates how a monolithic shopping flow can be decomposed into focused services for storefront, authentication, product catalog, cart, and orders.

This repository was refreshed from an older learning project and polished as a portfolio reference. The Kubernetes layer now highlights service discovery, private internal APIs, shared infrastructure services, runtime configuration, and safer deployment defaults.

## Architecture

```text
                Internet / local tunnel
                         |
                    LoadBalancer
                         |
                      store
                         |
    -------------------------------------------------
    |             |              |                  |
 auth-app    products-app     cart-app        orders-app
    |             |              |                  |
    ---------------- PostgreSQL ---------------------
                  Redis cache/session support
```

## Services

| Service | Role | Port |
| --- | --- | --- |
| `store` | Frontend-facing Django storefront | `8000` |
| `auth-app` | User registration, login, JWT, logout/blacklist checks | `8001` exposed internally as `8010` |
| `products-app` | Product catalog API and product pages | `8002` exposed internally as `8012` |
| `cart-app` | Cart API and product lookups | `8003` exposed internally as `8013` |
| `orders-app` | Order API and user/product enrichment | `8004` exposed internally as `8014` |
| `postgres` | Shared relational database for the Kubernetes demo | `5432` |
| `redis` | Cache/session support | `6379` |

## Kubernetes Improvements

The original manifests were generated by Kompose and worked as a first deployment draft. They have been rewritten to be easier to explain, safer to run, and better aligned with Kubernetes conventions.

- Replaced generated Kompose labels and embedded ConfigMaps with explicit app manifests.
- Removed ConfigMaps that contained source files and base64-encoded SQLite databases.
- Added a dedicated `microstore` namespace.
- Added shared runtime `ConfigMap` and `Secret` resources.
- Replaced hard-coded public service IPs with Kubernetes DNS service names.
- Exposed only the storefront through a `LoadBalancer`; backend APIs are `ClusterIP`.
- Added PostgreSQL and Redis manifests with stable image tags and resource limits.
- Added readiness and liveness probes.
- Switched application containers from Django `runserver` to Gunicorn.
- Moved database migrations into Kubernetes `Job` resources instead of running migrations in every replicated pod.
- Added `.dockerignore` files and stronger `.gitignore` rules for generated files, local databases, and secrets.

## Build Images

The Kubernetes manifests reference Docker images under `akreem/*`. If you fork or adapt this project, rebuild and push images with your own registry name:

```bash
docker build -t <registry>/micro_store:v1.7 ./store
docker build -t <registry>/micro_auth_app:v1.2 ./authentification
docker build -t <registry>/micro_products:v2.0 ./products
docker build -t <registry>/micro_cart_app:v1.0 ./cart
docker build -t <registry>/micro_orders_app:v1.0 ./orders
```

Then update the `image:` fields in `kubernetes/*-deployment.yaml` and `kubernetes/migration-jobs.yaml`.

## Deploy on Kubernetes

Apply the infrastructure and application manifests:

```bash
kubectl apply -f kubernetes/
kubectl -n microstore get pods,svc,jobs
```

For local clusters such as MicroK8s or Minikube, use a tunnel or port-forward if a cloud load balancer is not available:

```bash
kubectl -n microstore port-forward svc/store 8000:80
```

Open `http://localhost:8000`.

## Notes for Reviewers

This is intentionally a portfolio-scale microservices project, not a claim of production readiness. The current polish focuses on deployment hygiene and system design readability. Next production steps would include per-service databases, proper secret rotation, CI image publishing, ingress/TLS, observability, and replacing demo admin credentials before any public deployment.
